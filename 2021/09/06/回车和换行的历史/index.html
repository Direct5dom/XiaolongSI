回车和换行的历史

Posted on 2021-09-06 Edited on 2021-09-07 In Linux

不同操作系统的行尾符是有区别的：

  • Unix每行结尾为\n
  • Windows系统每行结尾是\r\n
  • Mac OS在 OS X以前每行结尾是\r， 现在每行结尾是\n.

本文将从历史角度去解释区别的原因所在。

概念区分

中文名 英文名 英文缩写 英文解释 C语言中表示 ASCII码
回车 Carriage Return CR return \r 0x0D
换行 Line Feed LF new line \n 0x0A

回车与换行的来历

机械打字机时代

早在计算机发明之前，机械打字机就已经走进了人们的生活。

这种老式打字机上有一个机械结构，名为Carriage，就是上图那个大圆柱体所在的结构。

在打字的时候Carriage会随着敲动键盘而向左移动，每输入一个字符，Carriage就向左前进一格，到了一行的最右边，Carriage也走到了最左边。

这时候需要执行两个动作，一个是将Carriage向右推回起始位置，这个动作叫做Carriage Return；另一个是将滚筒上卷一行，这个动作叫做Line Feed。

Carriage翻译成中文为“字车”，因此Carriage Return就翻译成了“回车”，而Line Feed就根据字面意思翻译成了“换行”。

电传打字机时代

仍然是在计算机没有出现之前，机械打字机的后继者电传打字机出现了，上述的“回车”、“换行”两个机械操作在电传打字机上仍然被沿用了下来。

在电传打字机上，手动的机械操作变成了换行符(LF)和回车符(CR)，从机械操作变成了字符输入，只是它没有任何真正的“字符”输入，背后的原理仍是机械操作。

个人认为后世计算机操作系统换行符问题的罪恶根源就在这里，到了电子时代明明能简化成一次操作的情况非要做成两个。

计算机时代

到了计算机时代，回车(CR)和换行(LF)作为两个“字符”进入了计算机的世界。电传打字机的CRLF操作也成为了标准进入了计算机时代，比如MS-DOS就是遵循CRLF标准的操作系统，这一特性从MS-DOS一直沿用至NT内核的Windows。

CRLF本应作为一个标准沿用到计算机时代里，像是很多互联网协议，如：RFC 0821 (SMTP)、RFC 1939 (POP)、RFC 2060 (IMAP)以及RFC 2616 (HTTP)。都指定CRLF作为行尾符。

但是实际上进入计算机时代后这个问题就变成了一个特别混乱的问题。

Unix采用LF作为行尾符，当然如果你查看stty选项，你将看到onlcr选项指定是否应将LF更改为CRLF。这个设置如果弄错了，会得到一个阶梯文本：

所以即便是Unix，在一开始也是使用CRLF作为行尾符。后来可能是为了空间成本的考虑（CRLF要比单纯的一个LF多占用一个字符的空间），Unix将行尾符定为LF。

而这便是真正噩梦的开始，后来大名鼎鼎的C语言在Unix时代的“祖先”也将Unix这一约定带入了C语言标准，也就是以\n(LF)作为行尾符这个规定。

后来C语言还引入了“换行符(newline)”一词来表达“generic line terminator”。并且ASCII委员会在1996年左右将字符0x0A的名称更改为“换行符(newline)”，这进一步让混淆程度提高。
